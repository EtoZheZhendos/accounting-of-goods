## 1. ОБЩИЕ СВЕДЕНИЯ

### 1.1. Обозначение и наименование программы

**Обозначение:** com.store.inventory-management  
**Полное наименование:** Система управления товарным учетом на складе  

### 1.2. Программное обеспечение, необходимое для функционирования программы

**Операционная система:**
- Windows 10/11 (64-bit)
- Linux (любой дистрибутив с поддержкой Java)
- macOS 10.14 и выше

**Системное программное обеспечение:**
- Java Runtime Environment (JRE) версии 17 или выше
- JavaFX Runtime версии 21

**Вспомогательное программное обеспечение:**
- Не требуется (все зависимости встроены)

### 1.3. Языки программирования

- **Основной язык:** Java 17
- **Язык разметки интерфейса:** FXML (XML-based)
- **Язык стилей:** CSS
- **Язык запросов:** HQL (Hibernate Query Language), SQL

### 1.4. Программа является составной частью

Программа является самостоятельным настольным приложением и не является составной частью другой системы.

## 2. ФУНКЦИОНАЛЬНОЕ НАЗНАЧЕНИЕ

### 2.1. Назначение программы

Программа предназначена для автоматизации складского учета товарно-материальных ценностей и включает следующие функциональные возможности:

**2.1.1. Ведение справочников:**
- Справочник номенклатуры товаров (артикул, наименование, производитель, единица измерения, минимальный запас)
- Справочник производителей (наименование, страна, контактная информация)
- Справочник складов (наименование, адрес, активность, структура хранения)

**2.1.2. Учет товарных позиций:**
- Регистрация товарных позиций с указанием количества, цены, партии, места хранения
- Отслеживание статуса товара (на складе, продано, зарезервировано, повреждено, просрочено)
- Контроль сроков годности товаров
- Учет серийных номеров (опционально)

**2.1.3. Документооборот:**
- Оформление документов поступления товаров
- Оформление документов реализации товаров
- Оформление документов перемещения товаров между складами
- Проведение документов с автоматическим изменением остатков
- Просмотр документов с детализацией позиций

**2.1.4. Отчетность и анализ:**
- Формирование отчета по остаткам на складах с фильтрацией
- Просмотр истории операций по товарным позициям
- Контроль товаров с низким уровнем запаса
- Контроль просроченных товаров

### 2.2. Класс решаемых задач

Программа решает задачи оперативного и аналитического учета в области складской логистики:
- Задачи оперативного учета движения товаров
- Задачи контроля товарных остатков
- Задачи анализа товарооборота
- Задачи документального оформления операций

### 3.1. Архитектура программы

Программа построена на основе многослойной архитектуры:

```
┌─────────────────────────────────────────────────┐

│  ┌────────────────────────────────────────────┐ │
│  │  FXML Views + Controllers                   │ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│           Business Logic Layer                  │
│  ┌────────────────────────────────────────────┐ │
│  │  Service Classes                            │ │
│  │  - ReportService                            │ │
│  │  - ReceiptService                           │ │
│  │  - SaleService                              │ │
│  │  - MovementService                          │ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│          Data Access Layer (DAO)                │
│  ┌────────────────────────────────────────────┐ │
│  │  Repository Pattern                         │ │
│  │  - GenericDao (базовый)                     │ │
│  │  - NomenclatureDao, ItemDao и т.д.          │ │

└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│              ORM Layer (Hibernate)              │
│  ┌────────────────────────────────────────────┐ │
│  │  Domain Models (Entities)                   │ │
│  │  - Nomenclature, Item, Document и т.д.      │ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│           Database (H2 Database)                │
└─────────────────────────────────────────────────┘
```

### 3.2. Алгоритмы

**3.2.1. Алгоритм проведения документа поступления:**
1. Начало транзакции БД
2. Создание документа со статусом DRAFT
3. Для каждой позиции документа:
   - Создание новой товарной позиции (Item)
   - Установка количества, цены, полки размещения
   - Установка статуса IN_STOCK
   - Сохранение в БД
4. Создание записи в истории операций (RECEIPT)
5. Изменение статуса документа на CONFIRMED
6. Расчет итоговой суммы документа
7. Фиксация транзакции (commit)
8. При ошибке — откат транзакции (rollback)

**3.2.2. Алгоритм проведения документа реализации (FIFO):**
1. Начало транзакции БД
2. Создание документа со статусом DRAFT
3. Для каждой позиции документа:
   - Получение списка доступных товаров на складе (сортировка по дате поступления FIFO)
   - Списание товаров в порядке поступления:
     - Если количество товарной позиции ≥ требуемому → полное списание
     - Если количество товарной позиции < требуемого → частичное списание, переход к следующей позиции
   - Изменение статуса товаров на SOLD
   - Уменьшение количества или установка в 0
4. Создание записей в истории операций (SALE)
5. Изменение статуса документа на CONFIRMED
6. Расчет итоговой суммы документа
7. Фиксация транзакции (commit)
8. При недостатке товара — откат транзакции и сообщение об ошибке

**3.2.3. Алгоритм проведения документа перемещения:**
1. Начало транзакции БД
2. Создание документа со статусом DRAFT
3. Для каждой выбранной товарной позиции:
   - Изменение привязки к полке (currentShelf) на целевую полку
   - Проверка, что целевая полка принадлежит целевому складу
4. Создание записей в истории операций (MOVEMENT)
5. Изменение статуса документа на CONFIRMED
6. Фиксация транзакции (commit)
7. При ошибке — откат транзакции (rollback)

### 3.3. Структура программных модулей

**3.3.1. Модуль доменных моделей (com.store.inventory.domain):**
- `Manufacturer` — производитель
- `Nomenclature` — номенклатура товаров

- `Shelf` — полка (место хранения)
- `Item` — товарная позиция
- `Document` — документ движения
- `DocumentItem` — строка документа
- `History` — история операций
- Перечисления: `DocumentType`, `DocumentStatus`, `ItemStatus`, `OperationType`

**3.3.2. Модуль доступа к данным (com.store.inventory.repository):**
- `GenericDao<T, ID>` — базовый класс для всех DAO
- `ManufacturerDao` — работа с производителями
- `NomenclatureDao` — работа с номенклатурой
- `WarehouseDao` — работа со складами
- `ShelfDao` — работа с полками
- `ItemDao` — работа с товарными позициями
- `DocumentDao` — работа с документами
- `DocumentItemDao` — работа со строками документов
- `HistoryDao` — работа с историей операций

**3.3.3. Модуль бизнес-логики (com.store.inventory.service):**
- `ReportService` — формирование отчетов
- `ReceiptService` — обработка документов поступления
- `SaleService` — обработка документов реализации
- `MovementService` — обработка документов перемещения

**3.3.4. Модуль представления (com.store.inventory.controller):**
- `MainWindowController` — главное окно приложения
- `NomenclatureDialogController` — диалог номенклатуры
- `ManufacturerDialogController` — диалог производителя
- `WarehouseDialogController` — диалог склада
- `ReceiptDocumentController` — диалог документа поступления
- `SaleDocumentController` — диалог документа реализации
- `MovementDialogController` — диалог документа перемещения
- `DocumentViewDialogController` — просмотр документа
- `ItemHistoryController` — история операций по товару
- `ItemDialogController` — диалог товарной позиции

**3.3.5. Утилиты (com.store.inventory.util):**
- `HibernateUtil` — инициализация и конфигурация Hibernate
- `DataSeeder` — наполнение БД тестовыми данными

## 5. ВЫЗОВ И ЗАГРУЗКА

### 5.1. Способ вызова программы

**Вариант 1: Запуск через Maven (разработка):**
```bash
cd /path/to/project
mvn javafx:run
```

**Вариант 2: Запуск JAR-файла (после сборки):**
```bash
java -jar inventory-management-1.0.jar
```

**Вариант 3: Запуск через ярлык (Windows):**

```
javaw.exe -jar "C:\Program Files\InventoryManagement\inventory-management-1.0.jar"
```

### 5.2. Загрузка программы

**Этапы загрузки:**
1. Инициализация Java Virtual Machine (JVM)
2. Загрузка библиотек JavaFX
3. Инициализация Hibernate и подключение к БД H2
4. Создание схемы БД (при первом запуске)
5. Загрузка тестовых данных (DataSeeder, при первом запуске)
6. Инициализация главного окна приложения (MainWindow.fxml)
7. Отображение интерфейса пользователю

**Время загрузки:** 3-5 секунд на рекомендуемой конфигурации

### 5.3. Особые условия применения

- Программа должна иметь права на запись в каталог с БД
- При первом запуске создается файл `inventory.mv.db` в корневой директории проекта
- Не требуется установка дополнительного ПО (кроме Java Runtime)

## 6. ВХОДНЫЕ ДАННЫЕ

### 6.1. Организация входных данных

**6.1.1. Справочные данные:**

**Производитель:**
- Наименование (строка, обязательное, до 255 символов)
- Страна (строка, до 100 символов)
- Контактная информация (строка, до 500 символов)

**Номенклатура:**
- Артикул (строка, обязательное, уникальное, до 50 символов)
- Наименование (строка, обязательное, до 255 символов)
- Описание (текст, до 1000 символов)
- Единица измерения (строка, обязательное, до 20 символов: шт, кг, л, м)
- Производитель (ссылка на справочник)
- Минимальный запас (целое число, 0-1000)

**Склад:**
- Наименование (строка, обязательное, до 255 символов)
- Адрес (строка, до 500 символов)
- Активен (логическое значение: да/нет)

**Полка:**
- Код (строка, обязательное, формат: "A-1-5", до 20 символов)
- Склад (ссылка на справочник)
- Активна (логическое значение)

**6.1.2. Документы:**

**Документ поступления:**
- Номер документа (строка, уникальное, до 50 символов)
- Дата документа (дата, обязательное)
- Склад (ссылка на справочник, обязательное)
- Поставщик (строка, до 255 символов)
- Табличная часть (массив строк):
  - Номенклатура (ссылка, обязательное)
  - Количество (число с 3 знаками после запятой, > 0)
  - Цена (число с 2 знаками после запятой, ≥ 0)
  - Полка (ссылка, обязательное)
  - Номер партии (строка, до 50 символов)
  - Серийный номер (строка, до 50 символов)
  - Срок годности (дата)
- Примечания (текст, до 2000 символов)

**Документ реализации:**
- Номер документа (строка, уникальное)
- Дата документа (дата, обязательное)
- Склад (ссылка на справочник, обязательное)
- Покупатель (строка, до 255 символов)
- Табличная часть (массив строк):
  - Номенклатура (ссылка, обязательное)
  - Количество (число, > 0)
  - Цена продажи (число, ≥ 0)
- Примечания (текст)

**Документ перемещения:**
- Номер документа (строка, уникальное)
- Дата документа (дата, обязательное)
- Склад-источник (ссылка, обязательное)
- Склад-приемник (ссылка, обязательное)
- Табличная часть (выбор конкретных товарных позиций)
- Целевые полки на складе-приемнике
- Примечания (текст)

### 6.2. Ограничения на входные данные

- Текстовые поля не должны содержать управляющие символы
- Числовые поля должны быть в допустимых диапазонах
- Даты не могут быть в будущем (для документов)
- Артикул номенклатуры должен быть уникальным
- Номер документа должен быть уникальным
- Количество товара должно быть положительным
- При реализации количество не должно превышать остаток на складе

### 6.3. Методы контроля входных данных

- Валидация на уровне UI (проверка заполнения обязательных полей)
- Валидация на уровне бизнес-логики (проверка бизнес-правил)
- Проверка ограничений целостности на уровне БД (foreign keys, unique constraints)
- Проверка типов данных (автоматически через Hibernate)

## 7. ВЫХОДНЫЕ ДАННЫЕ

### 7.1. Организация выходных данных

**7.1.1. Отчет по остаткам на складе:**

**Формат:** Таблица в главном окне приложения

**Структура:**
| Поле | Тип | Описание |
|------|-----|----------|
| Артикул | Строка | Артикул номенклатуры |
| Наименование | Строка | Название товара |
| Производитель | Строка | Название производителя |
| Количество | Число | Остаток на складе |
| Ед. изм. | Строка | Единица измерения |
| Склад | Строка | Название склада |

**Фильтры:**
- По артикулу (частичное совпадение)
- По наименованию (частичное совпадение)
- По складу (точное совпадение)

**Сортировка:** По артикулу, по умолчанию

**7.1.2. История операций с товарной позицией:**

**Формат:** Таблица в диалоговом окне

**Структура:**
| Поле | Тип | Описание |
|------|-----|----------|
| Дата операции | Дата и время | Момент выполнения операции |
| Тип операции | Строка | Поступление, Реализация, Перемещение и т.д. |
| Количество | Число | Изменение количества (+/-) |
| Документ | Строка | Номер документа-основания |
| Полка (откуда) | Строка | Исходная полка (для перемещений) |
| Полка (куда) | Строка | Целевая полка |

**Сортировка:** По дате (от новых к старым)

**7.1.3. Просмотр документа:**

**Формат:** Диалоговое окно с двумя частями

**Часть 1 — Шапка документа:**
- Номер документа
- Тип документа
- Дата документа
- Склад
- Контрагент (поставщик/покупатель)
- Статус документа
- Общая сумма документа
- Примечания

**Часть 2 — Табличная часть (позиции):**
| Поле | Тип | Описание |
|------|-----|----------|
| Артикул | Строка | Артикул номенклатуры |
| Наименование | Строка | Название товара |
| Количество | Число | Количество товара |
| Ед. изм. | Строка | Единица измерения |
| Цена | Число | Цена за единицу |
| Сумма | Число | Итоговая сумма позиции |
| Полка | Строка | Место размещения |

### 7.2. Методы контроля выходных данных

- Автоматический расчет итоговых сумм
- Форматирование числовых значений (2 знака после запятой для сумм, 3 для количества)
- Перевод перечислений на русский язык для отображения
- Проверка корректности данных перед выводом
- Обработка случаев отсутствия данных (вывод "-" или пустой таблицы)

### 7.3. Формат выходных данных

**Числовые данные:**
- Количество: формат "0.000" (три знака после запятой)
- Цена и сумма: формат "0.00 ₽" (два знака после запятой, символ рубля)

**Даты:**
- Формат даты: "dd.MM.yyyy" (например, 06.11.2025)
- Формат даты и времени: "dd.MM.yyyy HH:mm:ss"

**Текстовые данные:**
- Кодировка: UTF-8
- Перенос текста в многострочных полях
- Обрезка лишних пробелов

## 8. СООБЩЕНИЯ ОПЕРАТОРУ

### 8.1. Типы сообщений

**Информационные сообщения (INFO):**
- "Данные успешно сохранены"
- "Документ успешно проведен"
- "Данные обновлены"

**Предупреждающие сообщения (WARNING):**
- "Пожалуйста, выберите элемент из списка"
- "Заполните обязательные поля"
- "Недостаточно товара на складе для реализации"

**Сообщения об ошибках (ERROR):**
- "Ошибка при сохранении данных: [детали]"
- "Не удалось загрузить данные: [детали]"
- "Ошибка подключения к базе данных"

### 8.2. Формат сообщений

Все сообщения выводятся в виде диалоговых окон (Alert) с типом:
- **Информация** — зеленая галочка
- **Предупреждение** — желтый треугольник
- **Ошибка** — красный крест

### 8.3. Журналирование

Все события и ошибки записываются в лог-файл: `logs/application.log`

**Формат записи:**
```
YYYY-MM-DD HH:mm:ss [Thread] LEVEL Package.Class - Message
```

**Уровни логирования:**
- DEBUG — отладочная информация
- INFO — информационные сообщения
- WARN — предупреждения
- ERROR — ошибки
