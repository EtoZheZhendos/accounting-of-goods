# Техническое задание
## Система управления товарным учетом на складе

---

## 1. Общие сведения

### 1.1. Наименование системы
**Система управления товарным учетом на складе** 

### 1.2. Назначение системы
Система предназначена для автоматизации учета товарно-материальных ценностей на складах предприятия, включая приемку, отгрузку, перемещение товаров между складами и формирование отчетности.

### 1.3. Основание для разработки
Необходимость автоматизации процессов складского учета и повышения эффективности управления товарными запасами.

---

## 2. Назначение и цели создания системы

### 2.1. Назначение системы
- Автоматизация учета движения товаров на складах
- Контроль остатков товарно-материальных ценностей
- Формирование отчетности по складским операциям
- Отслеживание истории операций с товарами
- Управление справочной информацией

### 2.2. Цели создания системы
- Повышение точности учета товаров
- Сокращение времени на проведение складских операций
- Минимизация ошибок при учете товародвижения
- Обеспечение прозрачности складских операций
- Оперативное получение информации об остатках

---

## 3. Характеристика объекта автоматизации

### 3.1. Краткие сведения об объекте автоматизации
Объектом автоматизации является складское хозяйство предприятия, включающее:
- Один или несколько складов
- Систему хранения товаров (стеллажи и полки)
- Номенклатуру товаров различных производителей
- Документооборот по движению товаров

### 3.2. Бизнес-процессы
**Основные процессы:**
- Поступление товаров от поставщиков
- Реализация (продажа) товаров покупателям
- Перемещение товаров между складами
- Списание товаров
- Инвентаризация

**Справочная информация:**
- Номенклатура товаров
- Производители
- Склады и места хранения
- Контрагенты

---

## 4. Требования к системе

### 4.1. Требования к структуре и функционированию системы

#### 4.1.1. Управление справочниками

**Номенклатура:**
- Добавление, редактирование, удаление номенклатурных позиций
- Поля: артикул (уникальный), наименование, описание, единица измерения
- Привязка к производителю
- Установка минимального уровня запаса
- Возможность создания производителя непосредственно из формы номенклатуры

**Производители:**
- Ведение справочника производителей товаров
- Поля: наименование, страна производства, контактная информация

**Склады:**
- Управление складами предприятия
- Поля: наименование, адрес, признак активности
- Иерархическая структура: склад → стеллажи → полки
- Кодирование мест хранения (например, A-1-5)

#### 4.1.2. Учет товарных позиций

**Характеристики товарной позиции:**
- Привязка к номенклатуре
- Количество
- Цена поступления/реализации
- Номер партии
- Серийный номер (опционально)
- Срок годности (опционально)
- Место хранения (полка)
- Статус: на складе, продано, зарезервировано, повреждено, просрочено, возвращено

**Учетная политика:**
- Метод FIFO (First In, First Out) при реализации товаров
- Партионный учет товаров
- Контроль сроков годности

#### 4.1.3. Документы движения товаров

**Документ поступления:**
- Номер и дата документа (уникальный номер)
- Склад поступления
- Поставщик (контрагент)
- Табличная часть: номенклатура, количество, цена, полка размещения
- Автоматический расчет суммы документа
- Статусы: черновик, проведен, отменен

**Документ реализации:**
- Номер и дата документа
- Склад списания
- Покупатель (контрагент)
- Табличная часть: номенклатура, количество, цена продажи
- Автоматический выбор товаров по методу FIFO
- Проверка наличия товаров на складе

**Документ перемещения:**
- Номер и дата документа
- Склад-источник и склад-приемник
- Табличная часть: номенклатура, количество
- Выбор конкретных товарных позиций для перемещения
- Указание полок на складе-приемнике

**Общие требования к документам:**
- Все документы проводятся в рамках транзакции БД
- При проведении документа создаются записи в истории операций
- Изменение остатков происходит только при проведении документа
- Возможность просмотра документа с детализацией всех позиций

#### 4.1.4. Отчеты и аналитика

**Отчет по остаткам на складе:**
- Группировка по номенклатуре и складам
- Фильтрация по артикулу, наименованию, складу
- Отображение: артикул, наименование, производитель, количество, единица измерения, склад
- Возможность обновления данных

**История операций:**
- Просмотр всех операций с товарной позицией
- Отображение типа операции, даты, изменения количества, документа-основания
- Фильтрация по периоду, типу операции

**Контроль остатков:**
- Идентификация товаров с низким уровнем запаса
- Список просроченных товаров
- Список товаров с истекающим сроком годности

### 4.2. Требования к видам обеспечения

#### 4.2.1. Требования к программному обеспечению

**Технологический стек:**
- Язык программирования: Java 17+
- Фреймворк UI: JavaFX 21
- ORM: Hibernate 6.x
- База данных: H2 Database (встроенная)
- Система сборки: Apache Maven 3.x
- Логирование: SLF4J + Logback

**Архитектура приложения:**
- Паттерн MVC (Model-View-Controller)
- Слой представления: JavaFX FXML + Controllers
- Слой бизнес-логики: Service Layer
- Слой доступа к данным: DAO (Repository Pattern)
- Управление транзакциями: Hibernate Transaction API

#### 4.2.2. Требования к техническому обеспечению

**Минимальные системные требования:**
- Процессор: Intel Core i3 или аналог
- Оперативная память: 4 ГБ
- Свободное место на диске: 500 МБ
- Операционная система: Windows 10/11, Linux, macOS
- Разрешение экрана: 1280×720 пикселей

**Рекомендуемые требования:**
- Процессор: Intel Core i5 или выше
- Оперативная память: 8 ГБ
- Операционная система: Windows 10/11 (64-bit)
- Разрешение экрана: 1920×1080 пикселей

#### 4.2.3. Требования к информационному обеспечению

**Структура базы данных:**
- Таблицы: manufacturers, nomenclature, warehouses, shelves, items, documents, document_items, history
- Использование первичных и внешних ключей
- Индексы на часто используемых полях
- Ограничения целостности данных

**Хранение данных:**
- Файл базы данных: `inventory.mv.db`
- Автоматическое создание схемы при первом запуске
- Механизм миграций не требуется (H2 embedded)

### 4.3. Требования к интерфейсу пользователя

#### 4.3.1. Общие требования к UI
- Интуитивно понятный графический интерфейс
- Русскоязычный интерфейс
- Минималистичный дизайн
- Быстрый доступ к основным функциям через панель инструментов
- Табличное представление данных с возможностью сортировки

#### 4.3.2. Главное окно приложения
- Панель инструментов с кнопками быстрого доступа:
  - Обновить
  - Поступление
  - Реализация
  - Перемещение
  - Отчёты
- Вкладки для переключения между разделами:
  - Остатки на складе
  - Номенклатура
  - Документы
  - Склады
- Строка состояния для отображения статусной информации

#### 4.3.3. Диалоговые окна
- Модальные окна для редактирования данных
- Валидация вводимых данных
- Подсказки для обязательных полей (*)
- Кнопки: Сохранить, Отмена (или аналогичные)
- Возможность закрытия по Escape или клику вне окна

#### 4.3.4. Таблицы данных
- Сортировка по колонкам
- Изменяемая ширина колонок
- Выделение строки при выборе
- Контекстное меню (опционально)

### 4.4. Требования к защите информации

**Уровень защиты:**
- Для первой версии системы специальные средства защиты не требуются
- Доступ к данным имеют все пользователи системы
- База данных хранится локально в файловой системе

### 4.5. Требования к надежности

**Обработка ошибок:**
- Все исключения должны обрабатываться
- Пользователю выводятся понятные сообщения об ошибках
- Критические ошибки записываются в лог-файл

**Целостность данных:**
- Использование транзакций БД для всех операций записи
- Откат транзакции при возникновении ошибки
- Валидация данных перед сохранением

**Резервное копирование:**
- Возможность резервного копирования файла БД
- Ручное копирование файла `inventory.mv.db`

### 4.6. Требования к эргономике и удобству использования

- Логичная группировка функций
- Минимальное количество действий для выполнения типовых операций
- Подсказки (tooltips) для кнопок и полей
- Клавиатурные сокращения для часто используемых функций
- Автоматическое заполнение полей, где это возможно
- Запоминание последних выбранных значений в фильтрах
